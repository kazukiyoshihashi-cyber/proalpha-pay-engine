## 0. 本書について

この文書は、保険代理店における手数料・歩合給計算ソフトウェアの要件定義書です。リサーチに基づき、国内外の参考プロダクトの優れた部分を統合し、「安東さんの頭の中にしかない計算式」を形式知化してシステムに落とし込むことを最大の目的に設計しています。

Claude Codeでの使い方について一点補足します。Claude Codeはプロジェクトルートの CLAUDE.md を毎セッション自動読込しますが、公式ドキュメントおよびコミュニティのベストプラクティスでは、CLAUDE.md自体は20〜80行程度の短い「索引」に留め、本書のような長文の詳細仕様は `docs/requirements.md` のような別ファイルに置いてCLAUDE.mdから参照させる構成が推奨されています（長すぎるCLAUDE.mdは指示の一貫性が落ちるため）。そのため本書は単体のリファレンスドキュメントとして書いており、巻末（付録B）に短いCLAUDE.mdのサンプルを用意しました。

## 1. エグゼクティブサマリー

**課題：** プロアルファ様の手数料・歩合給計算は、安東さんという特定個人がExcel上に構築した計算ロジックに依存しています。これは事業継続性（安東さんが不在になった場合に誰も計算できない）、監査対応力（金融庁検査や税務調査で計算根拠を説明できるか）、スケーラビリティ（拠点や募集人が増えるほど手作業の負荷とミスのリスクが増す）の3つの観点でリスクです。

**ゴール：** 「安東さんが今Excelでやっていることを、安東さん以外の誰かが、コードを書かずに設定・検証・引き継ぎできる」計算エンジンを中核に据えたシステムを構築すること。単なる給与計算ソフトではなく、属人化した計算ロジックを資産化するナレッジマネジメントシステムとして設計します。

**アプローチ：** 前回リサーチで挙がった3つの参照軸（①保険代理店専業の手数料計算ソフト「サクッと明細」、②保険代理店CRM群の手数料管理機能、③国内給与ソフトの歩合給UI）に加えて、今回さらに踏み込み、海外のセールスインセンティブ管理（ICM/SPM）市場のトップランナーであるCaptivateIQ・Salesforce Spiff・Everstageが確立した設計思想——ノーコードのルールビルダー、Excel互換の数式エンジン、有効日付管理によるルールのバージョン管理、変更前のWhat-ifシミュレーション、計算根拠を遡れる監査証跡、自然言語からのルール生成支援——を統合しました。これらは「属人化したスプレッドシート計算を、コードを書かずに置き換える」という、まさに今回の課題と同じ問題に対する、年間数十億円規模の市場で磨かれてきた解法です。

**開発方針：** Claude Codeによる段階的プロトタイピングを前提とします。Phase 1はVercelで素早く動くデモを作り、安東さんとの対話を重ねながら計算ロジックを磨き込み、Phase 3でAWSへ本番移行するという2段階デプロイ戦略を採ります。さらに、プロアルファ様で磨き込んだ後、安東さんの評価を踏まえて他の保険代理店へ展開していく方針が確定したため、データモデルは当初からマルチテナント前提で設計します（詳細は第2.2章）。

## 2. 前提条件とスコープ

要件を絞り込むにあたり、いくつかの仮定を置いています。公開情報から推定した部分は明示し、第14章「確認事項」で改めて確認をお願いします。

### 2.1 対象代理店についての仮定（要確認）

公開されている情報によれば、株式会社プロアルファ様は大分県臼杵市を本拠地とし、中津支社・大分支社など複数拠点を展開する、創業70年以上の老舗代理店です。東京海上日動火災保険・東京海上日動あんしん生命保険のTOP QUALITY Ⅲ（最高ランク）認定を受けており、損害保険・生命保険を幅広く取り扱っています。この情報から、以下を仮定として置きます。

- 主要取扱保険会社は東京海上日動グループが中心（ただし他社商品も扱う乗合代理店である可能性があり、取扱保険会社数は要確認）
- 複数拠点（支社）があり、拠点ごとに業績・人員規模が異なる
- 募集人の雇用形態は雇用型（正社員・契約社員）が中心と推測されるが、業務委託型（個人代理店・法人代理店）が混在している可能性も否定できない（第5章で詳述する通り、保険業界はこの区分が報酬計算に直結するため重要な確認ポイント）

### 2.2 事業モデルとスコープ

**事業モデルの前提（今回確定）：** 本システムは、まず株式会社プロアルファ様向けに構築し、安東さんご自身がその完成度を見極めた上で、他の保険代理店へ推薦していくという流れを伺いました。つまりプロアルファ様は「導入第一号」であり、安東さんは事実上のプロダクト検証者・社内外への推薦者を兼ねる立場になります。第二号以降の代理店への展開時は、各社固有の手数料計算ロジック（その代理店にとっての「安東さん」が持つ知見）を、コードを書き直すのではなくANDO Engineのノーコード機能で都度カスタマイズして導入する、というモデルです。

この前提は技術設計に直接影響します。「後から複数代理店対応に作り直す」のではなく、データモデルとANDO Engineのルール管理機構は、Phase 1のプロトタイプ段階から最初からテナント（代理店）単位で分離された設計にします（詳細は第7.10章・第9章）。一方で、テナント管理者向けの管理画面やオンボーディングフローのような「複数代理店を運用するための機能」自体は、実際に第二号代理店の契約が見えてきた段階（Phase 5）で作り込めば十分です。「データ構造は最初からマルチテナント、機能は段階的に拡張」という分離が、今回の更新のポイントです。

**対象範囲（IN SCOPE）**

- 保険会社からの手数料データの取込・整理
- 手数料配分・歩合給計算ロジックのルール化（ANDO Engine）
- 給与計算本体（固定給・歩合給・社会保険料・源泉徴収・年末調整）
- 給与明細・各種帳票の発行
- 基本的なダッシュボード・分析機能
- マルチテナント対応のデータ基盤（テナント間のデータ分離、テナントごとのANDO Engineルール独立管理）

**対象外（OUT OF SCOPE・将来検討）**

- 保険契約そのものの管理（顧客管理・契約管理・意向把握等のコンプライアンス機能）→ hokan等の既存代理店システムとのデータ連携を前提とし、本システムでは作り込まない
- 勤怠管理そのもの → 既存勤怠システムからのデータ取込を前提とする
- 保険募集行為の支援機能
- 課金・請求まわりの機能（複数代理店への商用提供時に必要になるが、本要件定義では対象外。Stripe等の外部サービス連携を将来検討）

**確定した将来オプション**

- 他の保険代理店への展開（マルチテナントSaaS化）：確定方針。プロアルファ様を磨き込みのリファレンスとし、安東さんの評価を踏まえて第二号以降に展開する。テナント管理者向け画面、白ラベル（代理店ごとのロゴ・ブランディング）対応、課金管理などは、実際の展開タイミングに合わせて段階的に追加する。

## 4. ドメイン理解：保険代理店の手数料・報酬構造（要件の前提知識）

なぜ既製の給与ソフトでは対応できないのか、その構造的な理由を整理します。これは安東さんへのヒアリングの「地図」としても機能します。

### 4.1 手数料は二段階構造である

保険代理店の収益構造は「保険会社→代理店」「代理店→個人（募集人）」という2段階の手数料配分でできています。

**保険会社→代理店：** 契約の年換算保険料 × 保険会社所定の手数料率。この料率は契約区分（新契約か継続か）、商品種別、さらに代理店の「ランク」や「品質係数」によって変動します。大手損保・生保各社は、取扱保険料の規模や契約継続率、コンプライアンス体制の評価点をもとに代理店をランク分けし、半年〜1年ごとに料率を見直す仕組みを持っています。

**代理店→個人（募集人）：** 代理店が受け取った手数料のうち何%を個人に配分するかは、各代理店が独自に定めるものです。業界平均は6〜7割程度とされますが、勤続年数・資格・実績ランクによって個人ごとに異なる配分率を設定している代理店も少なくありません。

つまり最終的な個人の歩合給は「年換算保険料 × 保険会社料率 × 代理店ランク係数 × 個人配分率」という、最低でも4つの可変要素の掛け算になります。さらに、これらの係数表自体が保険会社・商品・年齢帯・通貨などで多次元化されているケースもあります（例：外貨建て保険の継続手数料は、通貨・契約年齢帯・積立利率の水準によって料率が変わる多次元テーブルになっていることが一般的です）。

### 4.2 新契約手数料と継続（維持）手数料

生命保険は契約初年度に手数料が大きく支払われ（年間保険料の30〜100%、商品によっては150%超）、2年目以降は維持のための継続手数料（年間保険料の1〜5%程度）に切り替わるのが典型です。損害保険は逆に、新契約への偏重が小さく、契約継続のたびに毎年手数料が発生する「ストック型」収益構造を持ちます。この「新契約」と「継続」の区分を商品ごとに正しく判定し、異なる計算式を適用する必要があります。

### 4.3 早期解約によるクローバック（遡及減額）

契約者が一定期間内（13ヶ月以内など）に解約した場合、保険会社は代理店への支払済み手数料の一部または全部を取り戻します。これが個人の歩合給にも連動して遡及的な減額（クローバック）として反映される設計になっていることが一般的です。「先月までプラスだった歩合が、解約のせいで今月マイナス調整される」というのは、属人的な計算でもっとも事故が起きやすいポイントの一つです。

### 4.4 雇用形態の複線化——ここが最大の複雑性ポイント

保険業界は2014年の金融庁指導を契機に、業務委託型（個人事業主）募集人の直接雇用化が進みましたが、現在もなお実務上は複数の形態が混在しています。

**雇用型（正社員・契約社員）：** 固定給＋歩合給の構成。歩合給込みの賃金から残業代の単価を再計算する必要があり（歩合給に対する残業代の支払いは労基法上必須）、社会保険料・所得税（給与所得の源泉徴収税額表）が適用されます。

**業務委託型（個人事業主・法人代理店）：** 歩合給部分は「外交員報酬」として扱われ、給与所得とは異なる源泉徴収計算（所得税法204条、同一人に対する1ヶ月の支払額から12万円を控除した残額に10.21%を乗じる、という給与所得とはまったく別の計算式）が適用されます。インボイス制度対応（適格請求書発行事業者番号の管理）も必要です。

**ハイブリッド型：** 実務上もっとも厄介なのがこのパターンです。固定給部分は「給与所得」として通常の源泉徴収を行い、歩合給部分は「事業所得（外交員報酬）」として204条の計算を別途行う、という按分処理が、社労士事務所の実務資料でも紹介される現実的な運用として存在します。

**この4.4が、まさに「安東さんの頭の中にしかない」部分である可能性が高いと考えます。** 一般的な給与計算ソフト（マネーフォワード クラウド給与、freee人事労務、給料らくだ等）は雇用型の給与所得計算には強いものの、外交員報酬の204条計算や、個人ごとに「この人は給与所得、この人は事業所得、この人は按分」という区分自体を切り替えながら運用する機能までは想定していません。ANDO Engineはこの区分を「個人ごとの設定項目」として最初から第一級の概念に据える必要があります。

## 5. システムコンセプトと全体アーキテクチャ

### 5.1 設計思想

ANDO Engineを中核に、「データ取込」→「計算・ルール層」→「給与・帳票出力」という3層構造を取ります。重要なのは、計算ロジックそのもの（料率テーブル、配分率、条件分岐）をコードではなくデータとして保存し、実行時に解釈する設計にすることです。これにより、料率改定や安東さんの新しい気づきがあった場合に、エンジニアの介入なしに（あるいは最小限の介入で）ルールを追加・修正できます。

```
① データ取込層
  ├── 保険会社別 手数料データ (CSV/Excel)
  ├── 勤怠データ (外部システム連携)
  └── 契約・募集人マスタ
         ↓
② ANDO Engine（計算・ルール層）
  ├── ルールリポジトリ
  │     ├── 料率テーブル
  │     ├── 配分率テーブル
  │     ├── 条件分岐ロジック
  │     └── 有効日付管理
  ├── 計算実行エンジン (数式評価 + 決定木評価)
  └── シミュレーション/監査
        ├── What-ifテスト
        ├── 過去実績との突合
        └── 計算根拠トレース
         ↓
③ 給与・帳票出力層
  ├── 給与計算本体 (社保/源泉徴収/年末調整)
  ├── 給与明細・支払調書
  ├── 法定帳票 (算定基礎届/月額変更届等)
  └── ダッシュボード

人間系プロセス：
  安東さんへのヒアリング (ルール形式知化) → ルールリポジトリへ
  承認ワークフロー → 計算実行エンジンへ
```

### 5.2 なぜこの構造か

海外ICM/SPMソフトの調査で一貫して見られたのは、「①データ取込」「②ルール定義（ノーコード）」「③計算実行」「④検証・監査」「⑤出力」を明確に分離するアーキテクチャでした。特に③と④の間に「本番反映前に必ずシミュレーションを通す」というステップを挟む設計は、CaptivateIQ・Spiffの両方に共通する思想であり、給与という「間違えたら即座に人間関係の問題になる」領域では必須の安全装置だと考えます。

| プラットフォーム運営者（将来） | 複数代理店（テナント）を横断的に管理する立場 | 新規代理店のオンボーディングが容易にできること。各テナントが完全にデータ分離されていることを技術的に保証できること |

## 9. データモデル概要

主要エンティティの一覧です（詳細なカラム定義はClaude Codeでの実装時にPrismaスキーマ等として具体化することを想定）。マルチテナントSaaS化が確定方針のため、**Tenant以外の全エンティティは `tenant_id`（Tenantへの外部キー）を持つことを前提**とします。

- **Tenant（代理店テナント）：** 代理店名、契約プラン、稼働状態、作成日。プロアルファ様がレコードとして存在する最初のテナントになる
- **Employee（従業員・募集人）：** tenant_id、氏名、拠点ID、雇用形態区分、入社日、資格情報、マイナンバー（暗号化）、銀行口座
- **Branch（拠点）：** tenant_id、拠点名、拠点ランク
- **InsuranceCompany（保険会社）：** tenant_id、会社名、データフォーマット定義（保険会社マスタ自体はテナント間で共通化できる可能性があるが、代理店ごとに契約内容が異なるためMVPではテナント単位で保持する）
- **Product（商品）：** tenant_id、保険会社ID、商品名、新契約／継続区分
- **CommissionRecord（手数料明細）：** tenant_id、保険会社ID、商品ID、契約者情報（最小限）、年換算保険料、契約日、手数料額、取込元データ参照
- **AllocationRule（配分率ルール）：** tenant_id、対象従業員ID／拠点ID／商品ID、配分率、有効開始日、有効終了日
- **CalculationRule（ANDO Engineルール本体）：** tenant_id、ルール名、数式定義（JSON形式で保存）、条件分岐定義、有効開始日、有効終了日、作成者、承認者、承認状態
- **LookupTable（参照テーブル）：** tenant_id、テーブル名、次元定義（通貨×年齢帯×経過年数等）、値マトリクス
- **PayrollRun（給与計算実行）：** tenant_id、対象期間、ステータス、実行者、確定日時
- **PayrollLine（給与明細行）：** tenant_id、従業員ID、PayrollRun ID、固定給、歩合給、控除項目、適用ルールのトレース情報（JSON）
- **AuditLog（監査ログ）：** tenant_id、操作者、操作日時、操作対象、変更前後の値

### 10.2 ANDO Engine の実装方式（最重要な技術選定）

調査の結果、計算エンジンの実装には大きく2つのアプローチがあり、それぞれ一長一短があります。

| 方式 | 概要 | メリット | デメリット・留意点 |
|---|---|---|---|
| **A：HyperFormula採用** | Excel互換の数式評価エンジン（TypeScript製、400以上の関数を実装）をそのまま埋め込む | Excelに慣れた安東さんがそのまま移行しやすい数式構文。実績豊富で機能が網羅的 | ライセンスがGPLv3。商用クローズドソースとして複数代理店に販売する場合は有償の商用ライセンス契約が必要。将来の他社展開が確定している以上、早期にライセンス方針を確定する必要がある |
| **B：自作DSL＋軽量ライブラリ採用** | 条件分岐は GoRules の ZEN Engine（JSON Decision Model、MIT系ライセンス、Rust製で高速、Reactのビジュアルエディタも公開）、純粋な数式計算は formulajs（MIT）や mathjs を組み合わせる | ライセンスがクリーンで商用展開の自由度が高い。決定木（どのルールを適用するか）と数式（実際の金額計算）を分離できる | 実装の初期コストはAより高め。高度なExcel関数が必要な場合は個別実装が必要 |

**推奨（今回の方針確定を踏まえて更新）：** 他代理店への展開が確定方針となったため、当初案の「Phase3まで判断を先送りする」という進め方は推奨しません。Phase 1のプロトタイプ構築の時点で、(a) HyperFormulaの商用ライセンスを早期に購入する、(b) 方式B（GoRules ZEN Engine＋formulajsなどMITライセンス構成）で最初から組む、のいずれかを選んでおくことを推奨します。

> ⚠️ ライセンス解釈については法律専門家への確認を推奨します。上記は技術的な観点からの整理であり、法的な最終判断はご自身で弁護士等の専門家にご確認ください。

いずれの方式でも、**「条件分岐ルーティング層」と「数式計算層」を最初からコード上で分離**しておくことは引き続き重要です。

### 10.3 AWS本番アーキテクチャの選択肢

| 方式 | 概要 | 適する場合 |
|---|---|---|
| AWS Amplify Hosting | Next.jsのSSRをそのままサポートし、Vercelに近い体験でAWSアカウント内にホスティングできる | スピード優先、インフラ管理の手間を最小化したい場合 |
| ECS Fargate＋RDS＋VPC構成（CloudFront・WAF・KMS・CloudTrail含む） | コンテナベースで、VPC内にDBを閉じ込め、KMSでの暗号化やCloudTrailでの監査ログを統制しやすい | マイナンバー等の特定個人情報を扱う本番システムとして、より高い統制・監査要件がある場合（**本プロジェクトはマイナンバーを扱うため、本番ではこちらを推奨**） |

いずれの方式でも、OpenNext（CDK経由でNext.jsをAWSにデプロイするOSSアダプタ）やSST（Serverless Stack）を使うとNext.jsのSSR機能を保ったままAWSへ展開しやすくなります。

```
Phase 0（〜7月）: 安東さんへのヒアリング・ルール形式知化
Phase 1（7月〜8月）: Claude Codeでのプロトタイプ構築（Vercel）
Phase 2（8月〜9月）: 実データ突合・検証（過去実績との誤差ゼロ確認）
Phase 3（9月〜10月）: AWS本番移行・セキュリティ強化
Phase 4（10月〜11月）: プロアルファ様で本番運用開始
Phase 5（11月〜12月）: 安東さんの評価を踏まえ第二号代理店へ展開
```

**Phase 0（最優先・並行着手を推奨）：** 安東さんへの構造化ヒアリング。本書の第4章をたたき台に、実際の計算パターンを洗い出します。理想的には、安東さんの過去6〜12ヶ月分の計算結果（Excel等）を「正解データ」として確保しておくことが、Phase 2の検証で決定的に重要になります。

**Phase 1：** Claude Codeで、ANDO Engineのコアロジックとモックデータによる給与明細生成までを、Vercel上で動くプロトタイプとして構築します。実データではなく、想定パターン（付録Aに記載）でまず動作確認します。

**Phase 2：** 実データを投入し、安東さんの手計算結果とシステムの計算結果を突き合わせます。誤差が出た箇所＝ヒアリングで拾いきれなかった暗黙知、として第7.3章のE-06機能を使って特定・追加登録していきます。

**Phase 3：** マイナンバー等の機微情報を扱う本番運用に向け、AWS（10.3節の方式）へ移行し、セキュリティ・監査要件を満たします。

**Phase 4：** プロアルファ様で本番運用を開始します。安東さんに実際の運用を通じてシステムを評価していただく期間でもあり、ダッシュボード等の付加機能もこの段階で拡充します。

**Phase 5：** 安東さんの評価・推薦をもとに、第二号となる保険代理店への展開を行います。第7.10章のテナント管理機能（オンボーディングフロー等）を実装し、新しい代理店固有の計算ロジックをANDO Engineでヒアリング・カスタマイズします。

## 12. 受け入れ基準・検証方法

1. **最重要の受け入れ基準：** 過去6ヶ月分（理想は12ヶ月分）の実績データについて、ANDO Engineの計算結果と安東さんの手計算結果が、全従業員・全期間で一致すること（誤差ゼロ、または明確に説明できる差異のみ）。これがUAT（受入テスト）の合格ラインです。

2. 計算根拠トレース機能（E-08）を使い、無作為に抽出した10件程度の明細について、安東さん以外の担当者（例：経理担当）が「なぜこの金額か」を画面だけで説明できること。

3. 最低賃金・割増賃金チェック機能が、意図的に作成した境界値テストケース（最低賃金ギリギリのケース等）で正しく警告を出すこと。

## 13. リスクと対応策

| リスク | 内容 | 対応策 |
|---|---|---|
| ヒアリング漏れ | 安東さんの暗黙知のうち、本人が「当たり前すぎて」言語化しない例外ルールが存在する可能性 | Phase 2の実データ突合で誤差が出た箇所を逆引きする運用を前提とし、ヒアリングを一度で終わらせようとしない |
| 保険会社側の手数料体系変更 | 各保険会社は半年〜1年ごとに料率・ランク制度を見直すため、追従コストが継続的に発生する | 料率テーブルをノーコードで更新できるANDO Engineの設計自体がこのリスクへの対応策になる |
| ライセンスリスク | HyperFormula等のOSSライセンスの解釈次第で、将来の商用展開に制約が生じる可能性 | 10.2節の通り、層を分離した設計で後からの差し替えを容易にしておく。将来の商用化が具体化した時点で専門家に確認する |
| 特定個人情報の漏洩 | マイナンバーを扱うことに伴う情報漏洩リスク | 暗号化・アクセス制御・監査ログを最初から要件に組み込み（S-01〜S-03）、本番はVPC内に閉じたAWS構成（10.3節）を採用する |
| 雇用形態の実態が仮定と異なるリスク | 第2.1節の仮定が外れた場合、報酬区分（P-04、P-05）の設計に手戻りが発生する | 第14章の確認事項を早期に確定させる |
| マルチテナント設計の手戻りリスク | プロアルファ様向けに単一テナント前提で作ってしまうと、後から複数代理店対応へ作り直すコストが大きい | データモデルの全主要テーブルにtenant_idを最初から組み込む（第7.10章・第9章）。Phase 1のプロトタイプ段階から複数テナントのダミーデータで動作確認を行う |

## 付録A：想定される計算パターン例（テストケース）

Claude Codeでの実装・検証時のテストケースとして、業界一般的に存在しうる計算パターンの例を示します（プロアルファ様固有のロジックではなく、あくまで業界一般のパターン例ですので、実際の安東さんのロジックとの照合が必要です）。

### パターン1：新契約手数料の基本配分

```
代理店受領手数料 = 年換算保険料 × 保険会社所定料率（商品マスタ参照）
              × 代理店ランク係数（品質取組係数 + 契約継続係数）
個人歩合給     = 代理店受領手数料 × 個人配分率（勤続年数・資格に応じテーブル参照）
```

### パターン2：継続（維持）手数料の多次元参照

```
継続手数料率 = LookupTable(通貨, 契約時年齢帯, 積立利率水準, 経過年数)
継続手数料額 = 積立金額 × 継続手数料率
```

### パターン3：早期解約クローバック

```
IF 契約からの経過月数 < 13 AND 解約 = true THEN
    調整額 = -1 × (既支給済み新契約手数料 × 0.5)
    調整対象期間 = 解約発生月の翌月給与
```

### パターン4：ハイブリッド型の報酬区分按分

```
給与所得部分の源泉徴収 = 源泉徴収税額表(固定給部分, 扶養人数)
事業所得（外交員報酬）部分の源泉徴収 = MAX(0, 歩合給部分 - 120,000) × 10.21%
最終手取り計算は上記2つを合算
```

### パターン5：拠点・紹介をまたぐ分配

```
IF 紹介者と契約担当者が別人の場合 THEN
    紹介料 = 代理店受領手数料 × 紹介料率（固定%、紹介者へ）
    成約担当者の歩合 = (代理店受領手数料 - 紹介料) × 個人配分率
ELSE
    成約担当者の歩合 = 代理店受領手数料 × 個人配分率
```

本書を `docs/requirements.md` に配置した上で、リポジトリルートに以下のような短いCLAUDE.mdを置くことを推奨します。

```markdown
# AlphaPay（保険代理店向け手数料・給与計算システム）

詳細要件は docs/requirements.md を参照。本ファイルでは要点のみ記載する。

## 最重要コンセプト
ANDO Engine（計算式・ルールエンジン）が本システムの心臓部。
属人化した手数料計算ロジックを、コードを書かずに設定・検証できることが最優先。

- DB: PostgreSQL + Prisma
- Phase1ホスティング: Vercel / Phase3: AWS
```
